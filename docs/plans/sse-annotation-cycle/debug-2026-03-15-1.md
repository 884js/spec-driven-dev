# 不具合調査: SSE Annotation Cycle のコメント待機〜読み取りが動かない

調査日: 2026-03-15

## 症状

パターン: 期待 vs 実際

- 操作: ブラウザでコメントを送信する
- 期待: CLI 側が即座にイベントを検知し、comments.json を読み取って writer に委譲する
- 実際: コメント送信後、ずっと待機中のまま。サーバー起動と SSE 待機が両方走っている状態

## 原因

### 原因1: SKILL.md の PORT 取得不備（SKILL.md:241-245）

サーバーを `Bash(run_in_background)` で起動後、「stdout から PORT を取得する」と記述されているが、`run_in_background` は即座にタスク ID を返し、PORT 出力は含まれない。`TaskOutput` で読み取る指示がないため、Claude が PORT を正しく取得できず、後続の `curl` コマンドが正しいエンドポイントに接続できない可能性がある。

### 原因2: curl|grep パイプラインの ~17秒遅延（SKILL.md:253-256）

```bash
curl -s -N http://localhost:{port}/api/plans/{feature}/events | grep --line-buffered -m 1 "^event: \(comments_saved\|review_done\)"
```

POSIX パイプラインのセマンティクスにより、shell は全プロセスの終了を待つ。grep がマッチして終了しても、curl はサーバーからの次のデータ受信まで recv() ブロック中のため、パイプへの write を試行しない。サーバーの SSE ハートビート間隔が15秒のため、curl が SIGPIPE を受信して終了するまで ~17秒かかる。

実測テスト（3回）で 16-17 秒を確認。

## 修正方針

### コード修正

#### server.py — SSE 接続の `?once=` パラメータ対応

- URL ルーティング正規表現をクエリパラメータ対応に変更（`events$` → `events` + クエリ分離）
- `_serve_sse` で `once` クエリパラメータを解析
- `once` で指定されたイベント名を送信後、接続を即座に閉じる（`break`）
- ブラウザの EventSource（`once` なし）は影響なし

#### SKILL.md — 2箇所修正

1. **PORT 取得**（L241-245）: `run_in_background` 後に `TaskOutput` で PORT を取得する手順を明記
2. **curl URL**（L253-256）: `?once=comments_saved,review_done` パラメータを追加

なし
