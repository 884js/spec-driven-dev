# 不具合調査: 複数タブレビュー時にコメントがプラン単位で反映されない

調査日: 2026-03-14

## 症状

パターン: 期待 vs 実際

- **操作**: 複数の `/spec` セッションを同時実行し、ブラウザの複数タブでレビュー
- **期待**: 1プランごとにコメント送信 → writer 修正 → 差分反映のサイクルが独立して動く
- **実際**: 全プランの「レビュー完了」を押すまでコメントが反映されない

## 原因

### 実行フロー

1. CLI A が `server.py` を起動（`run_in_background`）→ サーバーの stdout を所有 ← ✓ 正常
2. CLI B が `server.py` を実行 → 既存サーバーを検知 → `register_plan_remote()` → `return` で即終了（`server.py:226-228`）← ★ 問題
3. CLI B が `TaskOutput(block=true)` → プロセス既終了のため即座に返る ← ✗ イベント受信不可
4. ブラウザで feature-b のコメント送信 → サーバーが `EVENT:comments_saved:feature-b` を stdout に出力 ← CLI A の stdout に流れる
5. CLI A の SKILL.md は feature-a 用 → feature-b のイベントを処理しない ← ✗ 無視される

### 根本原因

マルチタブアーキテクチャにおいて、サーバーの **stdout イベント通知**はサーバープロセスを起動した CLI のみが受信できる設計になっている。2番目以降の CLI の `server.py` は register 後に即 `return` するため、イベントを受信する手段がない。

## 修正方針

### コード修正

1. **`scripts/annotation-viewer/server.py`**（`_finish_review` メソッド）:
   - `POST /api/plans/{feature}/finish` で `review-done.flag` ファイルを plan ディレクトリに作成するよう追加
   - stdout イベント出力は維持（ログ目的）

2. **`skills/spec/SKILL.md`**（Step 3-c イベントループ）:
   - `TaskOutput(block=true)` による stdout 待機を廃止
   - `comments.json` と `review-done.flag` の存在をファイルポーリング（2秒間隔）で監視するループに変更
   - 各 CLI が自分の plan ディレクトリのみを監視するため、マルチセッションでも独立動作する

### デバッグログの除去

なし（調査用ログは追加していない）
