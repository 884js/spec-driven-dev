# リサーチ: plan.md ブラウザプレビューのライブリロード対応

調査日: 2026-03-10
調査タイプ: 複合

## 調査ゴール
現在の「閉じて→修正→開く」ブラウザプレビューワークフローを、開きっぱなしでファイル変更がリアルタイムに反映される仕組みに改善できるか、実現方法を明らかにする。

## 現状

### 現在のアーキテクチャ
- `scripts/annotation-viewer/server.py`: Python 3 標準ライブラリの HTTP サーバー（120行）
- `scripts/annotation-viewer/viewer.html`: HTML/CSS/JS 単一ファイル（614行）
- ブラウザ側で marked.js + mermaid.js + highlight.js + jsdiff による Markdown レンダリング
- コメント UI（セクション単位・テキスト選択）、差分ハイライト、ダーク/ライトモード対応

### 現在の更新フロー
1. サーバー起動 → ブラウザ open
2. ユーザーがコメント送信 → `POST /api/done` → サーバーシャットダウン
3. スキルが comments.json を読み取り plan.md を修正
4. サーバー再起動 → ブラウザ再度 open（差分ハイライト付き）

## 調査結果

### 推奨: アプローチ A（既存サーバーにポーリング追加）

外部ツールに置き換えると、現在の annotation-viewer が持つカスタム機能（コメント UI、差分ハイライト、mermaid 対応）が全て失われる。既存の server.py + viewer.html に軽量なポーリング機能を追加するのが最も合理的。

### 実装方針

#### server.py への変更（約10行追加）
- 新エンドポイント `GET /api/check` を追加
- `plan.md` の `os.path.getmtime()` を返す
- レスポンス例: `{"mtime": 1710000000.0}`

#### viewer.html への変更（約15行追加）
- `setInterval` で 1〜2 秒ごとに `/api/check` をポーリング
- mtime が変わったら `/api/plan` を再 fetch してコンテンツを再レンダリング
- スクロール位置を保持（`scrollTop` を保存・復元）
- 再レンダリング時に mermaid + hljs も再実行

#### SKILL.md の更新
- Step 4-c のフローを変更: サーバー停止→再起動のループをやめる
- plan.md 修正後はサーバーを停止せず、ブラウザが自動で変更を検知
- コメント送信時のみサーバーを停止する現在の動作は維持

### 注意点
- ポーリング間隔は 1〜2 秒が妥当（CPU 負荷と応答性のバランス）
- コメント送信中のレンダリング競合を避けるため、コメントモード中はポーリングを一時停止する設計が望ましい
- `plan.md.bak` の差分ハイライトは初回ロード時のみ適用し、ポーリング更新時は最新内容のみ表示する（修正途中の中間状態での差分は不要）

## 推奨・結論

既存の server.py + viewer.html に **ポーリングベースのライブリロード**を追加するのが最善。実装量は server.py に約10行、viewer.html に約15行で、外部依存の追加なし。カスタム UI の全機能を維持したまま「開きっぱなしで自動反映」を実現できる。
