# リサーチ: plans データストアを SQLite DB に移行する可能性の調査

調査日: 2026-03-14
調査タイプ: 複合（コードベース + 外部技術）

## 調査ゴール

現在の md ファイルベースのプラン管理を SQLite に移行した場合のメリット・デメリット・実現可能性を明らかにする。

## 現状（コードベース調査）

### ファイル構成

各プランは `docs/plans/{feature-name}/` 配下にフラットに格納される（19プラン）。

| ファイル | 役割 | 生成タイミング |
|---------|------|--------------|
| `plan.md` | 機能仕様書 | /spec 完了時 |
| `progress.md` | 実装進捗管理 | /spec 完了時 |
| `result.md` | 検証結果 | /check 完了時 |
| `research-*.md` | 技術調査結果 | /research 実行時（複数可） |
| `debug-*.md` | 不具合調査記録 | /fix 実行時（複数可） |

### 各スキルの md 操作パターン

| スキル | 主な操作 | 対象ファイル |
|--------|---------|-------------|
| /list | 全プランの plan + progress + result を並列 Read → ステータス算出 | 全ファイル（O(3N)） |
| /spec | 全 plan.md を Glob 走査 → 関連プラン検出・並行開発チェック | 全 plan.md |
| /build | plan.md Read（読み取り専用）、progress.md Edit（状態更新） | 個別ファイル |
| /check | plan.md + progress.md Read → result.md Write | 個別ファイル |
| /fix | plan.md + progress.md Read → debug-*.md Write | 個別ファイル |
| /research | plan.md Read → research-*.md Write | 個別ファイル |

### フェーズ管理方式（ファイル存在ベース推論）

- plan.md + progress.md の存在 → spec 完了
- result.md の存在 + judgment フィールド → check の結果
- NEEDS_FIX → spec 更新モードをトリガー

### /list のステータス算出ロジック（7段階）

1. result.md + judgment=NEEDS_FIX → `要修正`
2. result.md + judgment=PARTIAL → `部分合格`
3. result.md + judgment=PASS → `検証済み`
4. progress.md の全タスク `✓` → `実装完了`
5. progress.md に `→` あり → `実装中`
6. progress.md 存在 + 全タスク `-` → `未着手`
7. plan.md のみ → `仕様作成済み`

### プラン間リレーション

- /spec Step 2 で全 plan.md を走査し、変更対象ファイルの重複・キーワード類似で関連候補を検出
- plan.md 本文中の Markdown テーブルに相対パスリンクで保存
- 双方向リンクはない

## 調査結果

### SQLite アクセス方法の制約

Claude Code のスキル/エージェントは .md ファイルであり、npm パッケージを直接 import できない。SQLite へのアクセスは **Bash ツール経由の sqlite3 CLI** に限定される。

### 選択肢の比較

| 観点 | SQLite（sqlite3 CLI 経由） | md ファイル（現行） |
|------|--------------------------|-----------------|
| 構造化クエリ | SQL で柔軟にクエリ可能 | Grep/Glob で代替（制限あり） |
| /list の速度 | 1つの SELECT 文で全プラン取得可能 | O(3N) の Read 呼び出し |
| リレーション管理 | FK + JOIN で自然に表現可能 | 本文中のテーブルに手動リンク |
| 人間による直接閲覧 | sqlite3 CLI か GUI ツールが必要 | テキストエディタで即閲覧可能 |
| 人間による直接編集 | SQL の知識が必要 | テキストエディタで即編集可能 |
| Git 差分 | バイナリのため差分不可（カスタム diff ドライバーで緩和可能） | テキスト差分がそのまま見える |
| 自由記述テキストの格納 | シングルクォート・改行・bash 特殊文字のエスケープ処理が必須 | そのまま記述可能 |
| 並行アクセス | WAL モードで読み書き並行可能（書き込みは直列） | ファイル単位で独立、競合リスク低 |
| エージェントの実装複雑度 | Bash + SQL + エスケープ処理 | Read/Write/Edit で直感的 |

### SQLite が有利な点

1. **/list の高速化**: 全プランのメタデータを1つの `SELECT` で取得可能。現行の O(3N) Read が O(1) になる
2. **リレーション管理**: 関連プラン・タスク依存関係を FK + JOIN で自然に表現。双方向リンクも容易
3. **集計・フィルタ**: ステータス別集計、日付範囲検索、特定ファイルに関連するプラン検索などが SQL で簡潔に書ける
4. **データ整合性**: スキーマ制約（NOT NULL, UNIQUE, FK）で不正データを防げる

### SQLite が不利な点（致命的な問題を含む）

1. **自由記述テキストの格納が極めて煩雑**（致命的）
   - plan.md の本文は任意の Markdown テキスト。シングルクォート、バックティック、`$` 変数記号、`#` 見出しなどを含む
   - Bash + SQLite の二重エスケープが必要。エージェントの .md 実装でこれを正確に行う信頼性が低い
   - 例: `it's done` → SQL 側で `it''s done` + Bash 側でもクォート処理

2. **人間の閲覧・編集体験の劣化**（重大）
   - 現行: ファイルを開くだけで仕様が読める。GitHub UI でも PR 上でも読める
   - SQLite: sqlite3 CLI や GUI ツールが必要。「ちょっと確認」のコストが大幅に上がる

3. **Git との相性の悪さ**（重大）
   - バイナリのため `git diff` で差分が読めない
   - PR レビューで変更内容を確認できない
   - カスタム diff ドライバー（`sqlite3 .dump` でテキスト変換）で緩和可能だが、マージは依然困難
   - `.gitattributes` + `.git/config` の設定を全開発者に展開する管理コスト

4. **Claude Code ツールとの親和性の低さ**
   - 現行: Read/Write/Edit は Claude Code のネイティブツールで、エラーハンドリングも安定
   - SQLite: Bash ツール経由で SQL を文字列として渡す。構文エラーや型不一致のデバッグが困難

5. **ファイルベース推論の設計思想との衝突**
   - プロジェクトルール: 「state.json は使わない。ファイルの存在でフェーズを推論する」
   - DB 化するとこの設計思想を根本的に変更することになる

### 規模感の現実

- 現在 19 プラン。仮に 100 プランになっても Read は数十回レベル
- Claude Code のツール呼び出しオーバーヘッド（数百ms/回）を考慮しても、/list の体感速度は数秒程度
- SQLite の構造化クエリの利点が活きるのは数千〜数万レコード規模

### ハイブリッド案の検討

md ファイルを正（human-readable、Git-friendly）として維持しつつ、メタデータのみを SQLite にキャッシュする案:

| 観点 | 評価 |
|------|------|
| /list の高速化 | メタデータ SELECT で高速化可能 |
| 人間の閲覧性 | md ファイルが正なので維持 |
| Git 差分 | md ファイルが正なので維持 |
| 実装コスト | md 書き込み + DB 同期の二重管理が必要 |
| 整合性リスク | md と DB の不整合が発生しうる |
| 複雑度 | 現行の約2倍の実装・保守コスト |

→ 二重管理の複雑さが利点を相殺する。現在の規模では過剰。

## 推奨・結論

**現行の md ファイルベースを継続することを推奨する。**

理由:
1. 最大の動機である `/list の高速化` は、現在の規模（19プラン）では実質的なボトルネックになっていない
2. SQLite 移行の最大の障壁（自由記述テキストの Bash + SQL 二重エスケープ）は、Claude Code プラグインの制約上、信頼性のある解決が困難
3. md ファイルの「開いてすぐ読める」「Git で差分が見える」という利点は、開発者体験として非常に価値が高い
4. プロジェクトの設計思想（ファイルベース推論）と整合する

**ただし、もし将来プラン数が 100+ に達し、/list の速度が実際に問題になった場合**:
- md ファイルを正として維持しつつ、frontmatter のメタデータだけを JSON キャッシュファイル（`docs/plans/.cache.json`）に書き出す軽量キャッシュ方式が現実的
- これなら Git diff も可能で、二重管理の複雑度も最小限

## 次のステップ

- /list の速度が実際にボトルネックになるまでは現行方式を継続
- プラン数が増えて問題が顕在化した場合、JSON キャッシュ方式を検討
