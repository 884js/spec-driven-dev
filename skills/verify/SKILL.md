# 仕様検証（Verify）

実装コードを **直接読み取り**、plan.md との乖離を双方向で検出する。
verifier エージェントが実コードと仕様書を突合し、検証の副産物として result.md（最終仕様書）を生成する。

入力: `docs/plans/{feature-name}/plan.md` + 実装コード（git diff or ファイル一覧）
出力: 検証レポート + `result.md`（検証結果から生成）

**パスルール**: `docs/plans/{feature-name}/` はカレントディレクトリ直下。`{feature-name}` は英語の kebab-case（例: `campaign-notification`）。パス区切り不可

## ワークフロー

```
Step 0: plan.md + 変更範囲の特定 + スコーピング
Step 1: 仕様と実装の直接突合（ドメイン別・重要度付き）
Step 2: ファイル構成の突合
Step 3: 受入条件の検証
Step 4: result.md 生成 + 検証レポート + 次のアクション提示
```

---

## 重要度レベル

全ステップの検出結果に以下の重要度を付与する:

| レベル | 基準 | 例 |
|--------|------|-----|
| **Critical** | 仕様の主要機能が未実装、または根本的に異なる | 仕様に記述された操作が丸ごと未実装、必要なデータモデルが未作成、受入条件の未充足 |
| **Warning** | 仕様との部分的な不一致。動作はするが仕様通りではない | 仕様の入出力要件の部分的な不足、バリデーションルール漏れ、テストなし |
| **Info** | 軽微な差異。仕様の意図は満たしているが記述と異なる | 命名の微差、実装アプローチの違い（結果は同等）、仕様外の追加実装 |

---

## Step 0: plan.md + 変更範囲の特定

スキル起動直後に、対象 feature を特定する:

```
Glob docs/plans/**/plan.md
```

$ARGUMENTS に feature-name が指定されている場合はそのディレクトリを使用。
複数の仕様書ディレクトリがある場合はユーザーに選択を求める。

```
Read docs/plans/{feature-name}/plan.md
```

### 0-a. progress.md によるスコーピング

```
Read docs/plans/{feature-name}/progress.md
```

progress.md が存在する場合、タスク進捗テーブルから検証範囲を決定する:

- **全タスク `✓`** → **フル検証**: 全セクション + 受入条件を検証
- **一部タスク `✓`** → **部分検証**: 完了タスクの対象ファイル・関連セクションのみ検証。未完了分は「検証対象外」と明示
- **全タスク `-`** → 検証対象なし。「実装が開始されていません」と案内して終了

progress.md が存在しない場合はフル検証として扱う（strategy を経由していないケース）。

### 0-b. 実装差分の取得

デフォルトブランチを動的に検出し、差分を取得する:

```
Bash: git symbolic-ref refs/remotes/origin/HEAD | sed 's@^refs/remotes/origin/@@'
```

上記で取得したブランチ名（例: main, master, develop）を {base-branch} として使用。
コマンドが失敗した場合は main をフォールバックとする。

```
Bash: git diff {base-branch} --name-only
```

差分がない場合は、ユーザーに比較対象のブランチ/コミットを確認する。

### 0-c. troubleshoot ドキュメントの読み込み

```
Glob docs/plans/{feature-name}/troubleshoot-*.md
```

troubleshoot ファイルが存在する場合:
1. 各ファイルの「原因」「修正方針」セクションを Read し、以下を抽出:
   - 各調査で特定された根本原因
   - 決定された修正内容
   - 調査間の連鎖（仮説の変遷）
2. 抽出結果を troubleshoot コンテキストとして保持（Step 1 の verifier と Step 4 のサマリ生成で使用）

troubleshoot ファイルが存在しない場合はスキップ。

### 0-d. 検証範囲の提示

```
plan.md と実装差分を読み込みました:
- 機能: {feature summary}
- 検証モード: {フル検証 / 部分検証（{完了タスク数}/{全タスク数}）}
- 検証対象セクション: {バックエンド / DB / フロントエンド — 該当するもの}
- 変更ファイル: {N}ファイル
- troubleshoot: {N}件（{あり / なし}）

検証を開始します。
```

## Step 1: 仕様と実装の直接突合

verifier エージェントが **実装コードを直接 Read** し、plan.md の各ドメインセクションと突合する。
summary ファイルへの中間依存を排除し、実コードと仕様の直接比較で正確性を確保する。

### 適応的実行

変更規模に応じて実行方法を切り替える（サブエージェントの起動コストを考慮し、少量のファイルでは一括処理の方が効率的なため）:

- **小規模**（変更ファイル 5 個以下、またはドメイン 1 つのみ）: 単一の verifier で全ドメインを一括検証
- **中〜大規模**（変更ファイル 6 個以上 & ドメイン 2 つ以上）: ドメイン別に並列実行

### ドメイン別パラメータ

該当するドメインごとに、以下のパラメータで verifier を起動する:

| ドメイン | plan.md セクション | 検証の焦点 |
|----------|-------------------|-----------|
| API（バックエンド） | バックエンド変更 | 仕様に記述された「どんな操作を提供するか」「入出力は何か」「エラーケース」を、実装コードが機能的に満たしているか |
| DB | DB変更 | 仕様に記述された「どんなデータを保持するか」「データ間の関係性」「ビジネス上の制約」を、実装コードが機能的に満たしているか |
| フロントエンド | フロントエンド変更 | 仕様に記述された「どんな画面/UIが必要か」「ユーザーが行える操作」「表示するデータ」を、実装コードが機能的に満たしているか。ワイヤーフレームがある場合はレイアウトの意図の反映も確認 |

### verifier プロンプト（共通テンプレート）

各ドメインの verifier は以下のテンプレートで起動する。中〜大規模の場合、各ドメインは互いに独立しているため並列で実行する。

```
Task(subagent_type: implementation-verifier):
  プロンプト: 「仕様と実装の突合検証を行ってください。

  チェック観点: {ドメイン}
  仕様書: docs/plans/{feature-name}/plan.md の「{plan.md セクション}」セクション
  実装ファイル: {対象ファイル一覧}

  注意: 仕様書は機能仕様書であり、具体的な実装名（エンドポイント、テーブル名、コンポーネント名等）は含まれていません。
  {検証の焦点}を検証してください。
  検証結果に加え、実装の主要ロジックをフロー別のシーケンス形式で報告してください（静的な仕様は箇条書き、動的な処理フローは Mermaid flowchart でトリガー→判断→結果を図示）。
  検証中に気づいた注意点（仕様通りだが留意すべきエッジケースや暗黙の前提等）があれば、検証メモとして併せて報告してください。
  不一致の報告や主要ロジックの記述では、ファイル名や関数名ではなく「何ができる・できない・どう変わった」という機能・振る舞いの観点で記述してください。技術的な詳細（ファイル名、関数名等）は補足として付記してください。
  仕様からの変更点には、ユーザーから見た振る舞いが変わるものだけを含めてください。振る舞いが同じ内部実装変更は検証メモに記載してください。
  ロジック > 仕様セクションでは実装手順や API 呼び出しではなく、判定基準・閾値・制約・結果を書いてください。

  {troubleshoot ファイルがある場合、以下を追加}
  troubleshoot コンテキスト:
  この機能では実装中に {N} 件の不具合調査が行われました。
  {各 troubleshoot の「原因」と「修正方針」の要約}

  仕様にない実装を検出した場合、troubleshoot に起因するかを判定し、
  理由列に記載してください。」
```

## Step 2: ファイル構成の突合

Step 1 がファイル内容の仕様適合を検証するのに対し、Step 2 は git diff ベースでファイル一覧の網羅性を検証する。

plan.md の各ドメインセクションのファイル情報と、git diff の変更ファイル一覧を突合する。

**部分検証時**: 完了タスクの対象ファイルのみを突合対象とする。

**チェック項目**:
- 仕様書で「新規」とされたファイルが実際に作成されているか
- 仕様書で「変更」とされたファイルが実際に変更されているか
- 仕様書に記載のないファイルが変更されていないか（意図しない変更の検出）

**重要度の適用**:
- 仕様のファイルが未作成/未変更 → Critical
- 仕様外のファイルが変更 → Info（リファクタリングや依存更新の可能性）

**出力**:
```
## ファイル構成の突合

| 仕様書のファイル | 操作 | 実際の状態 | 重要度 | 判定 |
|---------------|------|---------|--------|------|
| {ファイルパス} | 新規 | ✓ 作成済み | - | OK |
| {ファイルパス} | 変更 | ✗ 未変更 | Critical | NG |
| {ファイルパス} | - | 変更あり | Info | 要確認 |
```

## Step 3: 受入条件の検証

plan.md の受入条件を1つずつ検証する。

**フル検証時**: 全受入条件を検証。
**部分検証時**: 完了タスクに関連する受入条件のみ検証。plan.md のテスト方針にトレーサビリティ表があればそれを参照して対応を判定する。

**検証方法**:
- 各受入条件について、対応するテストコードまたは実装コードを Grep/Read で確認
- テストコードがある場合: テストの内容が受入条件を満たしているか確認
- テストコードがない場合: 実装コードから受入条件の充足を判断

**重要度の適用**:
- 実装もテストもなし → Critical
- 実装はあるがテストなし → Warning
- 実装もテストもあり → 重要度なし（PASS）

**出力**:
```
## 受入条件の検証

| # | 受入条件 | テスト | 実装 | 重要度 | 判定 |
|---|---------|-------|------|--------|------|
| 1 | {受入条件} | ✓ {テストファイル} | ✓ 実装済み | - | PASS |
| 2 | {受入条件} | ✗ テストなし | ✓ 実装済み | Warning | PARTIAL |
| 3 | {受入条件} | - | ✗ 未実装 | Critical | FAIL |
```

---

## Step 4: result.md 生成 + 検証レポート + 次のアクション提示

### 4-a. result.md の生成

Step 1 の verifier 出力（主要ロジック + 乖離理由）、Step 0-c の troubleshoot コンテキスト、
Step 3 の受入条件検証結果を統合し、`docs/plans/{feature-name}/result.md` を Write する。
**テンプレート**: `skills/verify/templates/result.md` を使用

### 4-b. 検証レポートの生成

Step 1〜3 の結果を統合し、最終判定を行う。

**判定の導出方法**:
各 verifier の不一致テーブルの重要度列から Critical / Warning 件数を集計し、Step 2・Step 3 の重要度も合算して最終判定を導出する。
verifier は PASS / NEEDS_FIX の2値で報告するが、NEEDS_FIX の内訳（Critical / Warning）から本スキルが3段階の最終判定を決定する。

**判定基準**:
- **PASS**: Critical 0件 & Warning 0件
- **PARTIAL**: Critical 0件、Warning のみ
- **NEEDS_FIX**: Critical 1件以上

**レポート出力**:
```
## 検証レポート

### 総合判定: {PASS / PARTIAL / NEEDS_FIX}

### サマリー
- 検証モード: {フル検証 / 部分検証（{完了タスク数}/{全タスク数}）}
- ファイル構成: {OK数}/{総数}
- バックエンド仕様: {PASS/PARTIAL/NEEDS_FIX}（Critical {N} / Warning {N} / Info {N}）
- DB仕様: {PASS/PARTIAL/NEEDS_FIX}（Critical {N} / Warning {N} / Info {N}）
- フロントエンド仕様: {PASS/PARTIAL/NEEDS_FIX}（Critical {N} / Warning {N} / Info {N}）
- 受入条件: {PASS数}/{総数}

### 検出事項（重要度順）

#### Critical
| # | 分類 | 方向 | 内容 | 対応方法 |
|---|------|------|------|---------|
| 1 | {分類} | 仕様→実装 | {内容} | {対応方法} |

#### Warning
| # | 分類 | 方向 | 内容 | 対応方法 |
|---|------|------|------|---------|
| 2 | {分類} | 実装→仕様 | {内容} | {対応方法} |

#### Info
| # | 内容 |
|---|------|
| 3 | {内容} |
```

該当する検出事項がないレベルのセクションは省略する。

### 4-c. 次のアクション提示

- **PASS の場合**: 「検証完了。PR作成/更新に進めます。」
- **PARTIAL の場合**:
  - 実装修正が必要 → 「以下の Warning 項目の修正を推奨します: {修正項目}」
  - 仕様修正が必要 → 「plan.md を直接編集して仕様を修正してください」
- **NEEDS_FIX の場合**:
  - 実装漏れ（仕様→実装） → 「以下の実装が不足しています: {不足項目}」
  - 仕様不足（実装→仕様） → 「plan.md を直接編集して仕様を更新してください」
  - 型不一致/ロジック差異 → 「仕様と実装に差異があります。どちらに合わせるか確認してください。」
- **検証メモがある場合**: verifier の検証メモを AskUserQuestion で提示し、対応したい項目をユーザーに選択してもらう
- **部分検証の場合**: 上記に加えて「残り {N} タスクの実装完了後、フル検証を実行してください。」
